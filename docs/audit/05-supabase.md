## Summary

- **The BubblesKit Supabase project is a *shared, multi-app* database**, not a clean per-template instance. It carries tables from at least **three different products**: the multi-agent chatbot (this repo), a **fitness/nutrition app** (food, recipes, exercises, routines, workouts, body logs — the `fit-chat` child), and an **older first-gen chat schema** (`conversations`/`messages`) that is now orphaned.
- **26 tables** in `public` (the old audit counted 14). **19 applied migrations** (old audit said 8). **2 active edge functions**. **3 public storage buckets**. **1 vector extension** + pgvector RAG functions for two separate domains.
- **The live migration history does NOT match the repo's `supabase/migrations/` files** — different version timestamps, and 11 of the 19 applied migrations have **no corresponding SQL file in this repo at all**. `supabase db push` from this repo would not reproduce this database.
- **The database is essentially empty** — only `chat_sessions` (2 rows) and `debug_traces` (1 row) hold any data; all 23 other tables are at 0 rows. This is a dev/template instance.
- **Security advisors flag 1 ERROR-class issue across 5 tables** (RLS disabled in a PostgREST-exposed schema), plus 11 WARN-class items. **Performance advisors flag ~65 items** (mostly cosmetic given zero traffic, but 10 genuinely unindexed FKs and 25 un-optimized RLS `auth.uid()` calls).

## Security Advisors (live `get_advisors security`)

### 🔴 ERROR — RLS disabled on PostgREST-exposed tables (5)
`food_items`, `food_serving_units`, `recipes`, `recipe_ingredients`, `intelligence_embeddings` are in the `public` schema (exposed via the REST API) with **RLS not enabled**. With the anon key, these are world-readable/writable. The four fitness tables are catalog/reference data (lower risk) but `intelligence_embeddings` holds RAG vector chunks that may encode source content.
→ Fix: `ALTER TABLE <t> ENABLE ROW LEVEL SECURITY;` + an appropriate read policy. [lint 0013](https://supabase.com/docs/guides/database/database-linter?lint=0013_rls_disabled_in_public)

### ⚠️ WARN (11)
1. **`rls_policy_always_true`** — `chat_messages.INSERT` (`Service inserts messages`), `debug_traces.INSERT` (`Service inserts traces`), `job_preferences.INSERT` + `.UPDATE` (demo) all use `WITH CHECK (true)` / `USING (true)`. Anonymous writes allowed; for chat/traces this should be a `service_role` check or session-ownership predicate. [lint 0024](https://supabase.com/docs/guides/database/database-linter?lint=0024_permissive_rls_policy)
2. **`function_search_path_mutable`** (6 functions) — `handle_new_user`, `search_intel`, `match_food_items`, `match_recipes`, `update_conversation_timestamp`, `touch_conversation_on_message` have no fixed `search_path`. Add `SET search_path = ''` (or `public`). [lint 0011](https://supabase.com/docs/guides/database/database-linter?lint=0011_function_search_path_mutable)
3. **`anon`/`authenticated_security_definer_function_executable`** — `handle_new_user()` is `SECURITY DEFINER` and callable by `anon`/`authenticated` via `/rest/v1/rpc/handle_new_user`. It's meant to fire as a trigger only; revoke `EXECUTE` from `anon`/`authenticated`. [lint 0028](https://supabase.com/docs/guides/database/database-linter?lint=0028_anon_security_definer_function_executable)
4. **`extension_in_public`** — `vector` (pgvector) installed in `public`. Move to a dedicated `extensions` schema. [lint 0014](https://supabase.com/docs/guides/database/database-linter?lint=0014_extension_in_public)
5. **`public_bucket_allows_listing`** — `chat-uploads` (see Storage above). [lint 0025](https://supabase.com/docs/guides/database/database-linter?lint=0025_public_bucket_allows_listing)
6. **`auth_leaked_password_protection`** — disabled. Enable HaveIBeenPwned check in Auth settings. [docs](https://supabase.com/docs/guides/auth/password-security)

### ℹ️ INFO — RLS enabled, no policy (5)
`agent_configs`, `agent_handoffs`, `agent_tools`, `agent_versions`, `tool_definitions` — locked to service-role only. Safe but fragile (the app relies on "no policy = no anon access"; one accidental policy opens writes). [lint 0008](https://supabase.com/docs/guides/database/database-linter?lint=0008_rls_enabled_no_policy)

## Performance Advisors (live `get_advisors performance`)

Most are low-impact given the DB has ~0 traffic, but they document real structural debt:

- **10 unindexed foreign keys** — `agent_handoffs.target_agent_id`, `agent_tools.tool_id`, `agent_versions.published_by`, `food_serving_units.food_item_id`/`recipe_id`, `intelligence_embeddings.entity_id`, `recipe_ingredients.food_item_id`/`recipe_id`, `user_memories.source_session_id`, `workout_logs.routine_id`. [lint 0001](https://supabase.com/docs/guides/database/database-linter?lint=0001_unindexed_foreign_keys)
- **25 `auth_rls_initplan` warnings** — RLS policies across `profiles`, `conversations`, `messages`, `reflections`, `body_logs`, `routines`, `workout_logs`, `user_goals`, `chat_sessions`, `chat_messages`, `user_memories`, `admin_roles`, `debug_traces` call `auth.uid()` (etc.) **per row** instead of `(select auth.uid())`. Cheap fix, real win at scale. [lint 0003](https://supabase.com/docs/guides/database/database-linter?lint=0003_auth_rls_initplan)
- **~30 unused indexes** — expected for a 0-traffic dev DB; revisit after real usage.

## Top risks

1. **🔴 Database is not reproducible from this repo.** 11 applied migrations have no SQL file; 8 more are version-drifted; 2 edge functions have no source; 3 buckets are created out-of-band. A fresh `supabase db push` would diverge from production. **This is the headline risk** — the "infrastructure as code" contract is broken.
2. **🔴 5 tables with RLS disabled** in a PostgREST-exposed schema (`intelligence_embeddings` + 4 fitness tables). Anon key can read them.
3. **🟠 Shared multi-app project with no isolation.** The template's chatbot tables, a fitness app, a demo, and a dead chat schema all live in one database with one anon key. Child-project schema leaks into the "template" instance. Decide: dedicated project per app, or per-app schemas + documented ownership.
4. **🟠 3 fully public storage buckets** with no size/MIME limits; `chat-uploads` is list-enumerable.
5. **⚠️ Anonymous INSERT** on `chat_messages`, `debug_traces`, and wide-open `job_preferences` demo (`true` policies).
6. **⚠️ `handle_new_user` is anon-callable** as a `SECURITY DEFINER` RPC, and 6 functions have mutable `search_path`.
7. **⚠️ `debug_traces.user_id` is TEXT** (not `uuid`), breaking FK joins and forcing `auth.uid()::text` casts.

## Top strengths

1. **Profile auth path is solid** — `handle_new_user` trigger + 3 own-row policies; `profiles` modeled identically across web/iOS/Android.
2. **RLS is enabled on 20 of 25 tables** with correct own-row predicates; the gaps are a finite, fixable list.
3. **Edge functions enforce `verify_jwt`** — both AI functions require a valid JWT.
4. **Good index coverage on user-owned tables** (FK + lookup indexes); the "unused index" noise is purely a zero-traffic artifact.
5. **pgvector RAG is wired for two domains** (`search_intel` for intel, `match_food_items`/`match_recipes` for fitness) with ivfflat indexes.
6. **Consistent conventions** where the app owns the schema — snake_case, `timestamptz`, `uuid` PKs, `ON DELETE CASCADE`, soft-delete flags.
